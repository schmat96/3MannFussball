Bis Meilenstein 4

Planung

Nach der Rückmeldung zum dritten Meilenstein wird besprochen was nun zu tun ist. Aus vergangenen Erfahrungen wird nun eine ausführlichere Aufgabenzuteilung erstellt; Die TODO's werden aufgelistet und fair auf alle Gruppenmitglieder verteilt.

Beispielsweise muss das Reconnect-System überarbeitet oder die Art und Weise, wie das Spiel gestartet wird, an die Projektanforderungen angepasst werden.

Features

Neue Features

  • Ingame Zeitanzeige
  • Optionen für den Gameleader in der Lobby
    • Spielzeit
    • Bots An/Aus
  • Online Spieler in einer Liste angezeigt
  • Spiel wird einfefroren, wenn ein Spieler die Verbindung verliert
  • Spiel wird beendet, wenn der Spieler nach 1 Minute nicht zurückkehrt
  • Systembenutzername wird als Benutzernamen vorgeschlagen

Vorgenommene Änderungen

  • Alte Spiele sind jetzt in einer Liste anstelle von Popups aufrufbar
  • Spieler und Ball sind jetzt etwas kleiner, um das Spiel etwas spannender zu machen

COMING SOON™!

  • Spieler werden kleiner, je näher sie am eigenen Tor stehen
  • Spieler werden langsamer, je näher sie am eigenen Tor stehen
Diese zukünftigen Änderungen sollen den allgemeinen Spielverlauf spannender machen.

Behind the Scenes

Im Hintergrund mussten einige Massnahmen ergriffen werden. Einserseits wurde das SVN-Repository aufgeräumt, andererseits wurde im Code selbst etwas aufgeräumt.

Beispielsweise mussten User-Objekte auf dem Server in einer HashMap anstelle einer ArrayList gespeichert werden, um Abmeldungen zu vereinfachen.

Qualitätsanforderungen

Man hat beschlossen, von nun an in regelmässigen Zeitabständen alle Methoden auf ihre Zeilenanzah zu überprüfen. Die Ergebnisse werden kurz analysiert und Klassen mit mehr als den erwünschten 30 Befehlszeilen werden überarbeitet; Bestenfalls, wenn möglich, vereinfacht, ansonsten auf mehrere Methoden aufgeteilt.

Die Resultate der jeweiligen Messungen werden auf diesem Blog zu sehen sein.

JUnit test

Bei den tests mussten wir uns mehrmals neu einigen, welche Klasse getestet werden sollte. Anfangs entschied man sich, die sogenannte Vektor-Klasse zu testen, welche Funktionen wie Vektorrotation und Vektoraddition bietet zu testen. Dies wurde jedoch vom Tutor abgeraten, da die Klasse nicht so sehr im Mittelpunkt des Projektes steht und eventuell etwas zu einfach zu testen wird.

Man entschied sich um für die Klasse Lobby, welche Spieler und laufende Spiele verwaltet. Diese wahl fürte zu einem weit grösserem Problem als zu erwarten war. Die Art der implementierung der Klasse entpuppte sich als grober Designfehler, da sowohl Server und Client ein Objekt derselben Klasse erstellten, jedoch jeweils verschiedene Konstruktoren und Funktionen benutzten. Nun musste vorerst die Klasse überarbeitet werden, bevor der JUnit-test ausgeführt werden konnte.

Bis Meilenstein 3

Timeline

Feedback vom MS 2

Nach dem Feedback von Filip Rinkmann wird das Projekt einmal überarbeitet. Als erstes wird das Netzwerkprotokoll in zwei enums aufgelistet (Commands, Options), welche über ein informatives Javadoc verfügen, in welchem angegeben ist, welcher Command welche Options und welche Argumente braucht.
Ausserdem wird im SVN Repository aufgeräumt.

Verknüpfung Server/Client

Das bisherige Server/Client Projekt wird mit der entstehenden GUI und Game Logik gekreuzt. Dabei wird festgestellt, dass ein UDP Protokoll gar nicht zwingend nötig ist. Folglich werden nun auch die Ingame Informationen über kleine Erweiterungen des bestehenden TCP Protokoll übergeben.
Das einzige Problem damit sind kleine Performanceeinbüssungen.

Man entscheidet, doch eine UDP-Logik zu erstellen, welche aber nicht Zeitgerecht zur Abgabe des 3. Meilensteins funktionieren will. Schlussendlich gibt man sich so weit zufrieden und nimmt sich vor, die Sache nach der Präsentation noch einmal genauer anzugehen.

Builddateien

Unterdessen wird mit dem JAR-Buildtool ANT gekämpft. Das Tool generiert mit einem XML-Anweisungsscript eine Ausführbare JAR-Datei, welche aber lange nur den Fehler "Could not find Main-Class" ausspuckt.

Über Hexagon

Hexagon ist ein Echtzeit-Multiplayer-Actionspiel, in welchem bis zu 3 Spieler versuchen, einen Ball jeweils in die gegnerischen Tore zu befördern.

Diese simplen Spielregeln sollen einen innovativen Spielablauf erzeugen:

  • Drei Spieler verteidigen auf einem regelmässig sechseckigen Spielfeld jeweils ein Tor an abwechselnden Seitenwänden.

  • Gespielt wird mit einem Ball, der zu beginn des Spiels und nach jedem gefallenen Tor von der Mitte aus in eine zufällige Richtung losrollt.

  • Gezählt werden nicht erzielte, sondern durchgelassene Tore. Das Spiel gewinnt, wer nach ablauf der Zeit die wenigsten Tore durchgelassen hat.

  • In der Nähe ihres eigenen Tores verkleinern sich die Spieler.

Hexagon ist ein Java-Opensource Projekt, welches im Rahmen der Vorlesung "CS108: Programmier-Projekt" an der Uni Basel umgesetzt wird. Das Projekt ist in 5 Perioden eingeteilt, die jeweils mit einer Präsentation (Meilenstein) enden. Dieser Blog ist eine Art Projekttagebuch, in welchem der Arbeitsfortschritt bis zu den jeweiligen Meilensteinen zusammengefasst wird.
